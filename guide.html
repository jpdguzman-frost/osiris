01

Getting Started

Connect, pick up context, start working. Under 5 minutes.

Every session follows the same three steps. Once you've done it once, it becomes muscle memory.

Step 1 — Connect Rex

Make sure the Rex MCP server is running in Claude Code. If you just opened a fresh session, type /mcp in Claude Code to check. You should see rex listed as connected.

PromptCheck the status of rex — is it connected? Give me the port so I can connect the Figma plugin.

Claude calls get_status and returns the port (e.g. 7880). Copy this.

Step 2 — Connect the Figma Plugin

  1. Open your Figma file and launch the Rex plugin.
  2. Paste the port from Step 1 (e.g. 7880) into the plugin's connection field.
  3. Hit connect. You should see "Connected" in the plugin panel.

That's it — Claude can now read and write to your Figma file. You can also start a two-way chat between the plugin and Claude Code.

PromptStart listening to my chat from the Figma plugin.

Claude calls wait_for_chat and will respond to anything you type in the Rex plugin chat panel.

Step 3 — Pick Up Where You Left Off

Before diving into new work, catch up on context. Rex stores design decisions as notes attached to the file. This is how you (and Claude) remember what happened last session.

Prompt — Every session startCheck the notes for this file — what were we working on? Give me a summary of recent decisions and any conventions I should know about.

Claude calls browse_notes and notes to pull back everything saved. This gives you and Claude shared context instantly.

Make this a habit. Starting every session with "check the notes" means Claude always has context. No more re-explaining what you're building or what decisions were made.

Step 4 — Take Notes As You Go

Notes are how Rex remembers. Anything important — a decision, a convention, something you rejected — save it. Future you (and your teammates) will thank you.

Save somethingRemember: we're using 8px grid spacing across all components. Padding is always multiples of 8.
Save a decisionTake note: we chose the sidebar navigation over bottom tabs. User testing showed people missed nav items with tabs.
Recall everythingWhat do you remember about this file? Show me all the notes.

That's it. Say "remember", "take note", or "save a note" and Rex stores it. Say "what do you remember" or "check the notes" and it pulls everything back. Simple.

Step 5 — Mimic a Screen from Osiris

Want to generate a design based on a reference from our database? One prompt.

PromptGenerate a screen from osiris "apple_22"

Claude automatically pulls the screen's structure from Osiris and builds it in your Figma file using Rex. No need to manually call SOM or chain tools — one prompt does the full pipeline.

You're Ready

From here you can start working. The rest of this guide covers specific tools and workflows you can explore as needed. But first — read the next section. It's the most important one.

How to Actually Use This

The most important section in this guide. Read this.

Rex and Osiris will not produce a polished design. You will — by iterating with them.

These tools generate studies, not finished work. The real value comes from the loop between the tool and the designer. Here's the workflow:

  1. Generate multiple studies. Ask Rex + Osiris to produce several variations. Don't settle on the first output. Ask for 3–5 different directions, mix references, combine concepts.
  2. Curate. Look at what was generated. Pick the elements that work — maybe the layout from one, the typography from another, the color approach from a third.
  3. Refine by hand. This is where design happens. The designer adjusts spacing, tweaks type, fixes alignment, adds the craft that makes it feel real. Rex can help execute, but you direct.
  4. Repeat. Generate more variations on your refined direction. Combine, curate, refine again. Each cycle gets closer.
  5. Benchmark & evaluate. When you're happy, use Osiris to score your design against the database. See where it stands. Close any gaps.
The tool is a collaborator, not a replacement. Rex + Osiris will get you to 70% fast. The last 30% — the polish, the nuance, the craft — comes from the designer iterating, refining, and making intentional choices. That's what makes the work good.

What this looks like in practice

Round 1 — ExploreGenerate 3 different dashboard layouts inspired by the top fintech screens in Osiris. Make them distinct — one minimal, one data-dense, one with a strong hero section. Put each on a separate frame.
Round 2 — CombineI like the layout from frame 1 and the card style from frame 3. Combine them into a new version. Keep the spacing from frame 1 but use the typography from frame 3.
Round 3 — After your manual refinementsI've refined the design manually. Take a screenshot and evaluate it against the fintech bucket benchmarks. Tell me where I'm strong and where I still need work.

Each round, the design gets sharper. The tool handles the heavy lifting. You handle the taste.

Generate. Curate. Refine. Repeat. That's how good work gets made.
02

What Is What

Two tools, two purposes, one design workflow.

Rex  Figma Control

Rex connects Claude Code directly to your open Figma file via a plugin. It can read your design (nodes, styles, variables, selections), write to it (create frames, set text, apply fills, build components), and remember design decisions via notes.

50+ tools. Think of it as Claude's hands inside Figma.

Osiris  Design Intelligence

Osiris is a curated database of analyzed UI screens. It provides scored references, benchmarks, similarity search, and a Mimic feature that generates Figma-ready designs from any reference screen.

12 tools. Think of it as Claude's design eye and reference library.

Prerequisites

Rex needs the Figma plugin running + the rex MCP server connected in Claude Code. Osiris just needs the osiris MCP server connected — no plugin, just internet access. Check both with /mcp.

03

Rex Basics

Reading & building in Figma — the operations you'll use daily.

Reading Your Design

Before Claude can help with your design, it needs to see it. These are the key read operations.

Get what's selected

PromptLook at what I have selected in Figma and describe the layout, spacing, and colors.

Claude calls get_selection to inspect your current selection, then describes what it finds.

Search for specific nodes

PromptFind all frames named "Card" on the current page and list their sizes.

Uses search_nodes with a name filter. You can also filter by type (FRAME, TEXT, INSTANCE, etc.).

Take a screenshot

PromptTake a screenshot of node 123:456 so you can see what it looks like.

screenshot captures a PNG of any node or the full page. Claude can then visually analyze it.

Inspect the full page

PromptGet me a summary of the current page — what's on it and how it's structured.

get_page returns the page tree. Use detail levels: summary, standard, or full.

Building & Editing

Create a frame with auto-layout

PromptCreate a card component: 360px wide, vertical auto-layout, 24px padding, 16px gap. Add a heading "Feature Title" in 20px bold and a paragraph of placeholder body text in 14px regular. White background, 8px corner radius, subtle drop shadow.

Claude uses create_node with nested children, auto-layout config, fills, effects, and text styling — all in one atomic call.

Update existing nodes

PromptChange the background of my selected frame to #1E1E1E and make all text inside it white.

Uses get_selection to find the node, then set_fills on the frame and batch_update_nodes on the text layers.

Set text content and styles

PromptSet the text on node 42:100 to "Welcome Back" in Inter Semi Bold 32px, color #111827.

set_text handles content, font loading, sizing, and color in one call. It supports mixed-style ranges too.

Clone and rearrange

PromptDuplicate the card I have selected 3 times and arrange them in a horizontal row with 20px spacing.

Uses clone_node repeatedly, then reparent_node or set_auto_layout on a parent frame to arrange them.

04

Rex Notes

Persistent design knowledge that survives across sessions.

Rex Notes are cloud-stored design decisions, conventions, and context. They help Claude (and your team) remember why things are the way they are.

Notes are your file's collective memory — decisions, conventions, and the reasoning behind them, shared with everyone in the same Figma file.

Saving a Note

PromptSave a note: "We always use 8px grid spacing. Padding should be multiples of 8 (8, 16, 24, 32). This applies to all components." Category: convention.

note stores this with category metadata. Notes are scoped to the current file — everyone working in the same Figma file sees them.

Available categories:

CategoryUse for
conventionReusable rules (spacing, naming, colors)
decisionDesign decisions with rationale
contextBackground info ("this page is for mobile onboarding")
rejectionWhat you tried and rejected, and why
relationshipHow things connect ("this card maps to the API /products endpoint")
preferenceStylistic preferences
correctionCorrections to previous notes or assumptions

Querying Notes

PromptWhat design conventions have been saved for this file? Check the notes.

notes searches by query text. browse_notes lists by category filter.

More Examples

Prompt — DecisionSave a decision note: "We chose Inter over SF Pro because this product targets web + Android, not iOS-only. Approved by design lead on March 10."
Prompt — RejectionSave a rejection note: "We tried a bottom tab bar on the dashboard but rejected it — user testing showed people missed the side nav items. Stick with sidebar navigation."
Prompt — CleanupRun a dry-run cleanup of notes — show me what would be removed without actually deleting anything.

cleanup_notes with dryRun: true previews stale or superseded notes.

When to use notes vs CLAUDE.md memory: Notes are for design knowledge — decisions about the UI, component rules, visual conventions. CLAUDE.md memory is for workflow preferences — how you want Claude to behave. Use both.

Components & Design Tokens

Building and managing your design system in Figma.

Working with Existing Components

Find and use a component

PromptSearch for a Button component in the file, then create an instance of the "Primary / Large" variant on the current page.

get_components → instantiate_component with variant properties.

Override instance properties

PromptOn my selected button instance, change the label to "Get Started" and set the icon to hidden.

set_instance_properties updates text, boolean, and instance-swap properties.

Creating Components

PromptConvert my selected frame into a component. Add these properties: - "Label" (text, default "Button") - "Icon Visible" (boolean, default true) - "Size" (variant: Small, Medium, Large) Set the description to "Primary action button for forms and CTAs."

create_component → add_component_property (for each) → set_description

Design Tokens / Variables

Set up a complete token system

PromptCreate a design token system: - Collection: "Colors" - Modes: "Light", "Dark" - Variables: - "bg/primary": Light=#FFFFFF, Dark=#111111 - "bg/secondary": Light=#F5F5F5, Dark=#1A1A1A - "text/primary": Light=#111111, Dark=#FFFFFF - "text/secondary": Light=#666666, Dark=#A0A0A0 - "accent/default": Light=#2563EB, Dark=#60A5FA

setup_design_tokens creates the entire system atomically — collection, modes, and all variables in one call.

Update token values

PromptUpdate the "accent/default" variable: set the Dark mode value to #818CF8.

update_variables modifies values for specific modes.

Inspect existing tokens

PromptList all variables in the "Colors" collection and show their values for each mode.

get_variables with collection name filter.

06

Rex Advanced

Batch operations, arbitrary code execution, and chat.

Batch Operations

When you need to update many nodes at once, batch tools are dramatically faster.

PromptFind all text layers on this page that use the color #333333 and change them all to #111827.

search_nodes → batch_update_nodes (up to 50 at a time). Claude will loop through in chunks if there are more.

PromptApply these changes atomically — if any one fails, roll back all of them: 1. Rename frame A to "Hero Section" 2. Change its background to linear-gradient (#1E40AF to #7C3AED) 3. Update the heading text to "Design at Scale"

batch_execute with atomic: true ensures all-or-nothing execution.

Execute (Escape Hatch)

For anything the dedicated tools don't cover, execute runs raw JavaScript in the Figma plugin sandbox.

PromptRun a script to find all frames wider than 1440px and log their names and widths.
Use execute sparingly. Prefer the dedicated tools — they have validation, error handling, and are easier to review. Only use execute for edge cases that no tool covers.

Chat (Plugin Communication)

Rex has a two-way chat system between Claude and the Figma plugin. You can also @mention Rex in Figma comments.

PromptListen for any incoming messages from the Figma plugin.

wait_for_chat long-polls for messages. send_chat_response sends replies back. send_chat_chunk supports streaming responses.

08

Mimic

Generate a screen in Figma from any reference in the Osiris database.

Mimic is the fastest way to go from a reference screen to a buildable Figma design. One prompt pulls the screen's structural blueprint from Osiris and builds it directly in your file.

The Basic Prompt

PromptGenerate a screen from osiris "apple_22"

That's it. Claude automatically fetches the screen's structure (called a SOM — Screen Object Model), then uses Rex to build it in your Figma file. No need to chain tools manually.

With a Target Size

PromptGenerate a screen from osiris "cheval-blanc_home_01" scaled to 1440x900.

Pass target dimensions and the structure is recalculated to fit.

Mimic Multiple for Studies

PromptGenerate screens from osiris: "stripe_pricing_01", "linear_dashboard_03", and "notion_home_02". Put each on a separate frame so I can compare them side by side.

This is the starting point for the generate → curate → refine workflow. Generate several, pick the best parts, then combine and polish.

What Happens Under the Hood

Behind the scenes, Osiris stores a SOM (Screen Object Model) for each screen — a JSON tree describing every element's type, size, position, colors, typography, spacing, and children. When you say "generate a screen from osiris", Claude:

  1. Calls osiris_get_screen_som to fetch the structural blueprint
  2. Translates each SOM node into Figma frames, text layers, fills, and auto-layout via Rex
  3. Builds it all in your current Figma page

You don't need to know this to use it. But if you want to inspect or save a SOM manually, the tools are there: osiris_get_screen_som and osiris_save_screen_som.

Remember: Mimic gives you a structural starting point, not a finished design. It's the first step in the generate → curate → refine loop. The designer's hand is what makes it real.
09

Scoring & Self-Evaluation

Benchmark your designs against the database.

Understanding the Rubric

PromptShow me the Osiris scoring rubric. I want to understand the exact criteria before evaluating my design.

osiris_get_scoring_rubric returns the dimensions and scales used to evaluate every screen in the database.

Key score dimensions (all 1–10):

DimensionMeasures
overall_qualityOverall design quality and polish
calm_confidentVisual calm, restraint, confidence
bold_forwardBoldness, forward-thinking energy
color_restraintDisciplined use of color
hierarchy_clarityClear visual hierarchy
glanceabilityHow quickly it communicates
densityInformation density balance
whitespace_ratioUse of breathing room
brand_confidenceHow strongly brand identity comes through

Get Benchmark Targets

PromptGet the benchmark scores for the "luxury" bucket — I want to know what my design should hit to be competitive.

osiris_get_bucket_benchmarks returns average quality, calm/confident, and bold/forward scores for the bucket.

Self-Evaluate and Compare

PromptI've self-scored my design: - overall_quality: 7 - calm_confident: 8 - bold_forward: 5 - hierarchy_clarity: 7 - whitespace_ratio: 8 Compare these against the luxury bucket benchmarks and tell me where I need to improve.

osiris_score_comparison returns a scorecard with MEETS/CLOSE/BELOW status for each metric, plus actionable gap analysis and recommendations.

The Evaluation Loop

  1. Get the scoring rubric to understand criteria
  2. Get bucket benchmarks for your target category
  3. Design (or iterate on) your screen
  4. Have Claude self-evaluate your design against the rubric
  5. Run the comparison against benchmarks
  6. Focus on the gaps identified
  7. Repeat until all metrics show MEETS

Combined Workflows

The most powerful patterns: reference, build, evaluate, iterate.

Workflow A: Research-Driven Design

Start with research in Osiris, then build in Figma with Rex.

Prompt — Full workflowI need to design a fintech dashboard. Here's what I want you to do: 1. Search Osiris for the highest-quality fintech dashboard screens 2. Get insights on what patterns work in this category 3. Pick the best reference and mimic it into my Figma page 4. Adapt it to our design tokens and component library 5. Take a screenshot when done so we can review

This prompt chains: osiris_search_screens → osiris_get_bucket_insights → mimic (SOM + Rex build) → screenshot

Workflow B: Mimic + Compare

Generate multiple references side by side, then pick the best parts.

PromptGenerate screens from osiris: "stripe_pricing_01", "linear_pricing_02", and "notion_pricing_01" — each at 1440x900. Put them side by side so I can compare approaches.

Three mimic'd screens in one prompt. Curate the best elements from each, then ask Rex to combine them into a new direction.

Workflow C: Score, Iterate, Verify

Use Osiris to evaluate, Rex to fix, Osiris to re-evaluate.

PromptTake a screenshot of my current design. Then: 1. Get the scoring rubric from Osiris 2. Self-evaluate my design against each dimension 3. Get the benchmark scores for the "premium-saas" bucket 4. Compare my scores vs benchmarks 5. For any dimension that scores BELOW, suggest specific changes 6. Make the changes in Figma using Rex 7. Take a new screenshot and re-evaluate

Workflow D: Find Inspiration, Save Knowledge

PromptSearch Osiris for calm, premium onboarding screens. Show me the top 5 with their scores. Then save a note summarizing the key patterns we should follow for our onboarding redesign — include the specific screens as references.

Combines osiris_search_screens → analysis → note (Rex) to capture insights as persistent team knowledge.

Prompt Patterns & Tips

Write better prompts, get better results.

Be Specific About Structure

Weak

"Make me a card."

Strong

"Create a card: 360px wide, vertical auto-layout, 24px padding, 16px gap between elements. White background (#FFFFFF), 12px corner radius, subtle drop shadow (0 2px 8px rgba(0,0,0,0.08)). Contents: 16:9 image placeholder at the top, then a title in 18px/600 Inter #111827, then a description in 14px/400 Inter #6B7280 with max 2 lines."

Reference What Exists

Good patternLook at my existing "Card - Feature" component, then create 3 new variants: one with an icon instead of an image, one with a CTA button at the bottom, and one compact version at 280px wide.

Claude inspects the existing component first, then builds variants that are consistent with it.

Chain Operations Explicitly

Good patternDo this step by step: 1. Search Osiris for luxury automotive hero sections 2. Show me the top 3 with scores 3. WAIT — let me pick which one to mimic before building anything 4. After I approve, generate the screen in Figma

Using numbered steps and explicit pauses ("WAIT", "show me before proceeding") keeps you in control of multi-step workflows.

Use Notes for Context

Good patternBefore making any changes, check the notes for any conventions about button sizing and color usage. Then apply those conventions as you build the new toolbar.

Ask for Screenshots at Checkpoints

Good patternAfter each major section (header, hero, features, footer), take a screenshot so I can review before you continue.
Tell Claude what you want the result to look like, not what tools to use. Say "build a pricing table with 3 tiers" rather than "use create_node to make frames."

Common Mistakes

MistakeFix
Not checking what's on the page firstStart prompts with "Look at my current page first, then..."
Giving hex colors without contextReference your design tokens: "use our bg/primary token" or give the hex
Forgetting to specify fontsAlways include font family, weight, and size for text
Building everything in one giant promptBreak complex layouts into sections with review checkpoints
Not saving decisions as notesAfter any design discussion, save the outcome as a note
